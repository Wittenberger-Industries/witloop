# OKF adoption — a knowledge format for wi's Markdown

wi already lives by one design principle above the rest: **state on disk, not in
context** — a `.wi/` folder of small Markdown files that phases hand off cheaply and
that survive a fresh context window. That is exactly the problem the [Open Knowledge
Format](https://github.com/GoogleCloudPlatform/knowledge-catalog/blob/main/okf/SPEC.md)
(OKF v0.1) sets out to standardize: knowledge as *a directory of Markdown files with
YAML frontmatter*, authored by people, generated by agents, read by both, diffable in
git, portable across tools and time.

This document records what we take from OKF and the exact profile wi adopts, so that
every file wi writes — the generated `.wi/` store and the plugin's own source docs —
follows one self-describing convention.

## 1. Why OKF fits wi

OKF's stated goals line up with wi's almost one-to-one:

| OKF goal | wi already does this as… |
|----------|--------------------------|
| A format **enrichment agents write into** | `scan`/`plan`/`ship` writing `.wi/` |
| Inform how **consumption agents read** it | every phase reading `.wi/` at low token cost |
| Facilitate **exchange** across systems | `.wi/` is committed, so the team + future self read it |
| Standardize a small set of **required** fields | wi has conventions but never pinned them down |

The last row is the gap. wi's files are *conventionally* structured but not
*self-describing*: nothing in `spec.md` says "I am a spec," so tooling and agents infer
type from path and filename. OKF closes that with one required field and a handful of
recommended ones — at almost zero cost, since it is just frontmatter on files we already
write.

What we actually learn from OKF, beyond the mechanics:

- **Self-description over convention.** A `type` field makes a file routable and
  filterable without path-sniffing. This is the one rule worth making mandatory.
- **Progressive disclosure via an index.** wi's `learnings.md` is already this idea; OKF
  generalizes it to a per-directory `index.md` so an agent can see what exists before
  opening anything. We extend the pattern to a root `.wi/index.md`.
- **Description-driven indexes.** OKF index entries reuse each concept's `description`
  frontmatter. That means indexes can be *generated*, not hand-maintained — less drift.
- **Permissive consumption.** OKF mandates that consumers tolerate missing files, unknown
  types, and broken links. wi's cross-references ("hooks") are sometimes
  not-yet-written; treating broken links as normal, not malformed, matches reality.
- **Portability framing.** An OKF bundle is a unit of exchange. A wi `.wi/` directory
  becomes a shareable knowledge bundle about a repo — not just wi's scratch state.

## 2. What OKF requires vs. recommends

OKF is deliberately minimal. A bundle is **conformant** (SPEC §9) if, and only if:

1. Every non-reserved `.md` file has a parseable YAML frontmatter block.
2. Every frontmatter block has a non-empty `type`.
3. Reserved files (`index.md`, `log.md`) follow their section structure when present.

Everything else is soft guidance, and consumers MUST NOT reject a bundle for missing
optional fields, unknown `type` values, unknown extra keys, broken links, or missing
`index.md`. That permissiveness is what lets the format survive refactors and partial
agent generation — so we adopt the hard rules strictly and treat the rest as defaults we
follow but never enforce against readers.

## 3. wi today vs. OKF (the gaps we close)

| Area | wi today | OKF | Action |
|------|----------|-----|--------|
| Frontmatter | only `SKILL.md` + `agents/*.md` (name/description) | every concept: required `type` | Add frontmatter to all concept docs + generated `.wi/` files |
| Self-typing | inferred from path | explicit `type` | Define a `type` vocabulary (§5) |
| Index | `learnings.md`, `adr/index.md` (tables) | `index.md`, bullet lists, `description`-driven | Keep them; add optional root `.wi/index.md`; pull descriptions from frontmatter |
| Log | `progress.md` `## Log` (flat, appended) | `log.md`, date-grouped, newest-first, bold keyword | Adopt ISO dates + bold-keyword in the Log; optional bundle `.wi/log.md` |
| Citations | ADR `## Sources` (optional) | `# Citations`, numbered | Standardize on `## Citations` (numbered) across docs |
| Links | relative + "hooks" | bundle-relative `/…`, tolerate broken | Prefer `/`-from-bundle-root; document broken links as normal |
| Version | none | `okf_version` in root index | Declare `okf_version: "0.1"` in `.wi/index.md` |

## 4. The wi OKF profile

This is the contract every wi-authored Markdown file follows.

### 4.1 Frontmatter

Every concept document opens with a YAML block. Required and recommended keys follow OKF;
wi adds a few domain extensions (OKF explicitly permits producer-defined keys, and
consumers preserve them).

```yaml
---
type: Spec                      # REQUIRED — see the vocabulary in §5
title: Add Stripe webhooks      # recommended — human display name
description: One-line summary used by index generators and previews.
timestamp: 2026-06-14           # recommended — ISO 8601 date (or datetime) of last meaningful change
tags: [payments, webhooks]      # optional — cross-cutting categorization
# wi extensions (optional, domain-specific):
goal: stripe-webhooks           # the goal slug a file belongs to
status: accepted                # for ADRs / progress / specs with lifecycle
resource: https://…             # canonical URI when the concept maps to a real asset
---
```

Rules:

- `type` is the only hard requirement. Pick a descriptive value from §5; never invent
  cryptic ones. Consumers tolerate unknown types, so adding a new one is safe.
- `description` is one sentence. It is what indexes and previews show — write it for a
  reader scanning a list.
- `timestamp` is ISO 8601. A bare date (`2026-06-14`) is fine; use a full datetime when
  ordering within a day matters.
- `resource` appears only when the concept describes a concrete external asset
  (`repo-map.md` → the repo URL; `orchestrator.md` → the Orchestrator folder URL). It is
  absent for abstract concepts (specs, pitfalls, glossaries).

### 4.2 Body

OKF's examples move the title into frontmatter and start the body at `#`-level section
headings. wi keeps a **body `# Title` H1 followed by `##` sections** — this is a
deliberate, conformant deviation: OKF body structure is soft guidance (§4.2), and an
explicit H1 reads well in plain Markdown and in GitHub. So: set `title:` in frontmatter
*and* keep the familiar `# Spec: <goal>` H1. No template's section layout changes.

Conventional section headings we use when applicable: `## Citations` (§4.5), and the
existing wi sections (`## Summary`, `## Acceptance criteria`, etc.) stay as they are.

### 4.3 Index files (`index.md`)

`index.md` is reserved: a directory listing for progressive disclosure, **no frontmatter**
(the bundle-root `.wi/index.md` is the one exception — it may carry `okf_version`).

- `.wi/index.md` *(new, optional but recommended)* — the root listing: the project-level
  files and a pointer to `goals/`. Declares `okf_version: "0.1"`.
- `.wi/learnings.md` — stays the learnings index (one line + hook per goal). It is a
  typed concept index, not the reserved directory listing; entries SHOULD reuse each
  learning's `description`.
- `.wi/adr/index.md` — stays a reserved index. We keep the **table** form (ID · Decision ·
  Status · Date · Goal): OKF §6 requires enumeration, not a specific bullet syntax, so a
  table is an acceptable deviation and is more useful for a decision log.

Entries should carry the linked concept's `description`, so an index can be regenerated
from frontmatter rather than hand-curated.

### 4.4 Log

OKF `log.md` is date-grouped (`## YYYY-MM-DD`), newest-first, each entry a prose line led
by a bold keyword (`**Update**`, `**Creation**`, `**Deprecation**`).

wi keeps the per-goal history in `progress.md`'s `## Log` (one writer, appended at each
phase transition). We adopt OKF's **ISO dates and bold-keyword** convention there. wi's
Log stays append-order (oldest-first) because it doubles as a resumable timeline for a
hands-off run — a documented, intentional deviation from newest-first. An optional
bundle-level `.wi/log.md` (newest-first) MAY record project-scope events.

### 4.5 Citations

When a body makes a claim sourced from outside the bundle, list sources under a
`## Citations` heading at the foot of the document, numbered:

```markdown
## Citations

[1] [UiPath REFramework docs](https://docs.uipath.com/…)
[2] [Internal data-quality runbook](https://wiki.example.internal/…)
```

This **renames the ADR template's `## Sources`** to `## Citations` and extends the
convention to any doc (spec, research, SDD) that leans on external material.
(`##`, not OKF's `#`, to sit under wi's body H1.)

### 4.6 Cross-links

- Prefer **bundle-relative** links from the `.wi/` root: `[customers](/tables/orders.md)`
  → for wi, `[the spec](/goals/stripe-webhooks/spec.md)`. These survive moving a file
  within its directory.
- Relative links (`./pitfalls.md`) are fine within a goal folder.
- **Broken links are not errors.** A link to not-yet-written knowledge is normal; never
  fail a run because a cross-reference is missing.

### 4.7 Conformance & versioning

A `.wi/` bundle is wi-OKF-conformant when every non-reserved `.md` has parseable
frontmatter with a non-empty `type`, and `index.md`/`log.md` follow their structure.
`scripts/validate.py` enforces this (parseable frontmatter + non-empty `type`) across the
repo's concept docs, in addition to the existing `name`+`description` check on executable
docs. The bundle declares the OKF version it targets via `okf_version: "0.1"` in
`.wi/index.md`.

## 6. Two file classes: knowledge vs. executable

wi's repo holds two kinds of Markdown, and OKF conformance applies a little differently:

- **Knowledge concepts** — references, docs, templates, and every generated `.wi/` file.
  These get the full §4.1 frontmatter. They are what OKF is for.
- **Executable docs** — `SKILL.md` and `agents/*.md`. These are the *consumer program*,
  not knowledge, and they carry a loader-required `name`+`description` contract on
  Claude Code, Codex, and Copilot. We make them conformant by adding `type` (`Skill` /
  `Agent`) **alongside** `name`/`description` — `name` serves as the OKF `title`,
  `description` as the OKF `description`. Extra frontmatter keys are tolerated by all
  three harnesses (the agents already carry `model`/`color`/`tools`). If a future harness
  ever rejects unknown keys, drop `type` from these two file classes and rely on the
  documented `name`→`title` mapping; nothing else depends on it.

## 7. Deliberate deviations (and why)

- **Body keeps an H1 title** (§4.2) — OKF puts the title only in frontmatter. We keep both
  for readability; conformant because body layout is soft guidance.
- **`## Citations`, not `# Citations`** (§4.5) — to sit under wi's body H1.
- **`adr/index.md` is a table** (§4.3) — OKF shows bullets; tables enumerate just as well
  and read better for a decision log.
- **`progress.md` Log is oldest-first** (§4.4) — it doubles as a resumable run timeline.
- **`learnings.md` is the named index, not `index.md`** — it is a specialized typed index;
  the reserved directory listing is the new `.wi/index.md`.
